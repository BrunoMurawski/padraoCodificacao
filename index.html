Sobre o Padrão AutoCode

O padrão AutoCode tem como finalidade a busca pela legibilidade, clareza e agilidade na compreensão, criação e comunicação dos códigos destinados a testes automatizados. Ele retrata algumas boas práticas de codificação com a finalidade de facilitar o entendimento do programador/testador ao ler o código, diminuindo assim a possibilidade de ocorrerem erros por má interpretação do mesmo. Esse padrão busca também obter uma melhor comunicação da equipe, encorajando a posse coletiva e almejando a produção e manutenção mais rápida em grupo através da através das boas práticas e o código limpo.

Boas Práticas - Clareza na Escrita

É comum no ambiente de desenvolvimento de software se trabalhar em equipe. Partindo dessa realidade, no cotidiano do programador, tão normal quanto escrever códigos são lê-los. Essa leitura pode ocorrer tanto na busca por novos conhecimentos na web, quanto no trabalho em um projeto em qualquer equipe.

Tendo isso em vista, é fundamental que o programador pense antes de escrever nossas classes e métodos, que outras pessoas irão lê-las e possivelmente realizar manutenções nas mesmas.

Por essa razão, este padrão recomenda que o programador ao escrever um trecho de código, pense em qual a percepção que possivelmente outros programadores teriam sobre o código escrito.

A seguir seguirão algumas recomendações que podem auxiliar na criação de um código mais intuitivo e de fácil manutenção:

Nomenclaturas

É muito importante que, ao se ler o nome de uma variável ou método seja possível entender o que ela/ele faz. Os nomes devem ser autoexplicativos, ser mais o completo possível e transmitir ao programador/testador sua razão de existir. Criar nomes intuitivos facilita a compreensão de seu código. Qualquer programador deve ser capaz de facilmente entender o que é proposto no contexto do código.

A quantia de tempo gasto lendo código versus escrevendo é bem mais de 10 para 1. Então, fazer o código mais fácil de ler, o torna mais fácil de escrever.

Robert C. Martin

Nomenclatura de Propriedades e WebElements:

Para todas as propriedades e WebElements deve ser adotada a escrita em língua inglesa utilizando o formato UpperCamelCase ou Pascal Case. Neste padrão todas as palavras devem iniciar com letra maiúscula e sem espaços. Em WebElements utilizados para os testes, devem ser adicionados prefixos para indicar que tipo de elemento é exibido na tela.

Ex: int CodigoItem { get; set; };.

Em WebElements utilizados para os testes, além de utilizar UpperCamelCase como formato, os mesmos devem indicar que tipo de WebElement são em seus nomes:

  • BtnNomeWebElement = Btn indica que o WebElement é uma botão exibido na tela
  • LblNomeWebElement = Lbl indica que o WebElement é uma label exibido na tela
  • LnkNomeWebElement = Lnk indica que o WebElement é uma link exibido na tela

Ex: IWebElement BtnLogarCarrinho{ get { return driver.FindElement(By.CssSelector(swBtnLogarCarrinho)); } }.

Nomenclatura de Variáveis e Atributos:

Para todas as variáveis e atributos de utilização interna deve ser adotada a escrita em língua inglesa utilizando o formato LowerCamelCase. Nesse formato, sempre a primeira letra é minúscula seguido pelas próximas palavras com a primeira letra maiúscula e sem espaços.

Ex: int quantidadeItensCarrinho;.

Caso o atributo referenciar uma propriedade o mesmo deve iniciar com underline"_"

Ex: int _codigoItem;.

Além de utilizar LowerCamelCase como formato, as variáveis e atributos devem indicar sua função em seu nome:

  • swNomeVariável = sw indica que a variável é uma String utilizada para guardar o identificador do WebElement
  • stNomeVariável = st indica que se trata de uma variável Static
  • _nomeAtributoIgualPropriedade = _ indica que o atributo referencia uma propriedade de mesmo nome

Nomenclatura de Constantes:

Todas as constantes definidas nos códigos devem ser escritas em língua inglesa e seguir o padrão UpperCase.

Nesse formato, todas as letra sempre são maiúsculas e como separador de palavras deve ser utilizado underline(_).

Ex: private final String NOME_DA_CONSTANTE.

Ex: private final String URL= "https://github.com/brunomurawski";

Nomenclatura de Classes:

Para a nomenclatura de classes também devemos manter a utilização da língua inglesa, mas utilizar como formato o padrão UpperCamelCase ou PascalCase. Neste padrão todas as palavras devem iniciar com letra maiúscula e sem espaços. Em classes de testes, devem ser adicionados o sufixo test.

Ex: class FormasPagamento{}

Ex Classes de Teste: class ComprarCamisasTest{}

Nomenclatura de Métodos:

Para a nomenclatura de métodos, como na de classes, também devemos manter a utilização da língua inglesa, mas utilizar como formato o padrão UpperCamelCase ou PascalCase. Neste padrão todas as palavras devem iniciar com letra maiúscula e sem espaços. Recomenda-se iniciar o nome do método com um verbo.

Ex: string GetFormaPagamento()

Ex: void LogarCarrinho()

Nomenclatura de Parâmetros:

Para a nomenclatura de parâmetros devemos manter a utilização da língua inglesa, mas utilizar como formato o padrão LowerCamelCase. Nesse formato, sempre a primeira letra é minúscula seguido pelas próximas palavras com a primeira letra maiúscula e sem espaços.

Ex: void LogarCarrinho( string usuario, string password)

Ex: double TotalItensCarrinho( string primeiroItem, string segundoItem)

Declaração de Variáveis e Atributos

O padrão indica a declaração das variáveis e atributos no topo da classe. Pois julga-se que as variáveis são melhores encontradas quando são declaradas no início do código. Tão logo o programador inicie a leitura do código verá os atributos e variáveis utilizados no contexto da classe.

Declaração Vertical:

O padrão AutoCode recomenda que as declarações de atributos, variáveis e propriedades devem ser feitas verticalmente.

Não é recomendado as declarações de atributos e variáveis horizontalmente. Declarando verticalmente, o programador conseguirá ler semanticamente e identificar mais rapidamente o código escrito.

Recomendado:

        
                private int quantidadeItensCarrinho;
                private int valorItem;
                private int quantidadeParcelasCartao;

Não Recomendado:

        
            private int quantidadeItensCarrinho, valorItem, quantidadeParcelasCartao;

Modificadores e Hierarquia de Declarações

O padrão indica que seja seguida uma hierarquia na declaração, ordenada pelos modificadores na seguinte ordem:

Public

Protected

Private

Essa ordem tem como objetivo demonstrar os atributos por nível de visibilidade. Indo desde a restrição de acesso mais aberta até a mais restrita:

Tabela de Modificadores:

Modifier Class Package Subclass World Obs
public ✔ ✔ ✔ ✔ (visível em qualquer lugar que use a classe na qual a classe foi declarada).
protected ✔ ✔ ✔ ✘ (visível na classe na qual a classe foi declarada e também em classes derivadas).
private ✔ ✔ ✘ ✘ (visível apenas na classe na qual a classe foi declarada).
no modifier ✔ ✔ ✘ ✘ (visível apenas na classe na qual a classe foi declarada, pois se o modifier não é declarado, o modifier o sistema aceita como private).

Recomendado:

        
            public class MinhaClasse {       //classe pública
                protected Decimal decimal;  //atributo decimal protected
                private int inteiro;       //atributo inteiro private
            }

Não recomendado:


           public class MinhaClasse {          //classe pública
                private int inteiro;          //atributo inteiro private
                protected Decimal decimal;   //atributo decimal protected
            }

Organização de Namespaces e Classes

O nome do Namespace deve refletir o propósito das classes presentes no mesmo:

Se o namespace contém as classes com chamadas a métodos de cenários de testes, o mesmo deve ter isso indicado em sua assinatura. Isso auxilia na compreensão do programador em relação a qual namespace a classe pertence.

Ex: ao se importar uma classe em um código, pode-se somente pela sua chamada na importação identificar a natureza de seu conteúdo.

Recomendado:

        
            using EcommerceFunctionalTests.Cenarios.Simular;
            namespace EcommerceFunctionalTests.Cenarios.Comprar.DoisCartoes.Logado.Busca

Não recomendado:

        
            using EcommerceFunctionalTests.Simular;
            namespace EcommerceFunctionalTests.Cenarios

Comentários

É recomendado a utilização de comentários somente quando há necessidade de uso. Deve-se evitar a utilização de comentários desnecessários e também a mistura de idiomas. Entretanto, caso a classe fique muito complexa devemos inserir no código alguns comentários sobre ela no formato XML.

Recomendado:


        ///< summary > Descrição da Classe ou Método
        ///< param > Descrever a origem dos parâmetros ou outras informações relevantes  < /param >
        ///< returns > Descrever o retorno do método < /returns >
        ///< exception cref="System.Exception" > Descrever a exceções tratadas < /exception > 
        ///< /summary >

Comentários em Commits

O padrão AutoCode recomenda a utilização de comentários em todos os commits efetuados para auxiliar na compreensão e resolução de problemas ocorridos em versões geradas.
