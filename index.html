Sobre o Padrão Better Way

O padrão Better Way tem como principal finalidade a busca pela legibilidade, clareza e agilidade na compreensão dos códigos. Ele retrata algumas boas práticas de programação com a finalidade de facilitar o entendimento do programador ao ler o código, diminuindo assim a possibildade de ocorrerem erros por má interpretação do código. O padrão almeja a produção e manutenção mais rápida em grupo através da através das boas práticas e o código limpo.

Boas Práticas - Clareza na Escrita

É comum no ambiente de desenvolvimento de software se trabalhar em equipe. Partindo dessa realidade, no cotidiano do programador, tão normal quanto escrever códigos são lê-los. Essa leitura pode ocorrer tanto na busca por novos conhecimentos na web, quanto no trabalho em um projeto em qualquer equipe.

Tendo isso em vista, é fundamental que o programador pense antes de escrever nossas classes e métodos, que outras pessoas irão lê-las e possivelmente realizar manutenções nas mesmas.

Por essa razão, este padrão recomenda que o programador ao escrever um trecho de código, pense em qual a percepção que possivelmente outros programadores teriam sobre o código escrito.

A seguir seguirão algumas recomendações que podem auxiliar na criação de um código mais intuitivo e de fácil manutenção:

Declaração de Variáveis e Métodos

O padrão indica a declaração das variáveis e atributos no topo da classe. Pois julga-se que as variáveis são melhores encontradas quando são declaradas no início do código. Tão logo o programador inicie a leitura do código verá os atributos e variáveis utilizados no contexto da classe.

Nomenclatura:

O padrão Better Way recomenda que as declarações de atributos e variáveis devem ser feitas verticalmente.

Não é recomendado as declarações de atributos e variáveis horizontalmente. Declarando verticalmente, o programador conseguirá ler semanticamente e identificar mais rapidamente o código escrito.

tse
private LoginPage login;
private Usuario usuario;
private CarrinhoPage carrinho;
private PesquisaProdutoPage pesquisa;
private ProdutoPage produtoPage;
private Produto produto;
private PesquisaProduto pesquisaProduto;
private Menu menu;
private MenuPage menuPage;

Ou:


public String sMenuCalcadosCrianca { get { return "#menu01 > li:nth-child(3)"; } }
public IWebElement MenuCalcadosCrianca { get { return driver.FindElement(By.CssSelector(sMenuCalcadosCrianca)); } }
        
public String sMenuCalcadosCriancaLink { get { return "#menu01 > li:nth-child(3) > div > div > ul > li:nth-child(3) > div > div > ul > li:nth-child(1) > a"; } }

O padrão indica que seja seguida uma hierarquia na declaração, ordenada pelos modificadores na seguinte ordem:

Public

Protected

Private

Essa ordem tem como objetivo demonstrar os atributos por nível de visibilidade. Indo desde a restrição de acesso mais aberta até a mais restrita

Modifier | Class | Package | Subclass | World

public | ✔ | ✔ | ✔ | ✔

protected | ✔ | ✔ | ✔ | ✘

no modifier | ✔ | ✔ | ✘ | ✘

private | ✔ | ✘ | ✘ | ✘

Recomendado:


public class MinhaClasse {    //classe pública
    private int inteiro;      //atributo inteiro private
    protected Decimal decimal;  //atributo float protected
    boolean ativado;          //atributo booleano namespace-private
}

Não recomendado:


public class MinhaClasse {    //classe pública
    private int inteiro;      //atributo inteiro private
    public Double valor;      //atributo público
    protected Decimal decimal;  //atributo float protected
    boolean ativado;          //atributo booleano namespace-private
}

Também é recomendado que atributos constantes como URL devem utilizar nomes "uppercase": private final String URL = "https://github.com/brunomurawski";

Declaração de Variáveis

Recomenda-se iniciar o nome da variável ou método com um verbo. Ser autoexplicativo, o mais completo possível e passar sua razão de existir pelo nome também é indicado. É muito importante que, ao se ler o nome da variável ou método seja possível entender o que ela/ele faz. Criar nomes intuitivos facilita a compreensão de seu código. Qualquer programador deve ser capaz de facilmente entender o que é proposto no contexto do código.

O padrão também recomenda que as palavras sejam separadas com letra maiúscula e colocar um nome que referencie o tipo de retorno e seja intuitivo

//código aqui com cenário descrito

Não recomendado:


//código aqui com nome genérico que não indique o que o cenário faz

Organização de Namespaces e Classes

O nome do Namespace deve refletir o propósito das classes presentes no mesmo:

Se o namespace contém as classes com chamadas a métodos de cenários de testes, o mesmo deve ter isso indicado em sua assinatura. Isso auxilia na compreensão do programador em relação a qual namespace a classe pertence.

Ex: ao se importar uma classe em um código, pode-se somente pela sua chamada na importação identificar a natureza de seu conteúdo.

Recomendado:


//using correto

Não recomendado:


//using incorreto

Comentários

É recomendado a utilização de comentários somente quando há necessidade de uso. Deve-se evitar a utilização de comentários desnecessários e também a mistura de idiomas. Entretanto, caso a classe fique muito complexa devemos inserir no código alguns comentários sobre ela.

Recomendado:


//using correto

Não recomendado:


//using incorreto
